Singularity's `Normal` background is `NONE` (transparent). The rose-pine convention of giving floats a *darker* `NormalFloat` background fails when the terminal background happens to match `bg_dark` â€” the float vanishes. The fix is **scoped**: only the **LSP hover popup** gets a lighter surface (`bg_soft = #353331`), applied via `winhighlight` on the hover window itself (`LspHoverNormal` / `LspHoverBorder` / `LspHoverTitle`). Global `NormalFloat` stays `bg_dark` so the file explorer, Telescope, Lazy and Mason keep their darker chrome. Initial implementation flipped `NormalFloat` globally; that turned the explorer lighter as a side effect and was reverted. Border style is `rounded`, applied globally via `vim.o.winborder` (Neovim â‰Ą 0.11) so every float shares one frame. Border colour is `orange` (`#E95420`) on `bg_dark` globally, with the hover popup overriding to `orange` on `bg_soft`.

Hover stays on native `vim.lsp.buf.hover` (no new plugin) with `max_width = 100`, `max_height = 30`, `wrap = true` â€” wider than the diagnostic float (80Ă—20) because TS hover output is denser (JSDoc + full type expansions). The hover surface override happens inside the `vim.lsp.util.open_floating_preview` wrapper, which sets `vim.wo[winid].winhighlight = "Normal:LspHoverNormal,FloatBorder:LspHoverBorder,FloatTitle:LspHoverTitle"` after the float is created.

**Signature help: no popup.** `hrsh7th/cmp-nvim-lsp-signature-help` was added then removed because it was the second source firing on `(`, recreating the two-popup overlap. `ray-x/lsp_signature.nvim` was tried as the single-source replacement but rendered text incorrectly and still showed two windows in practice (its float plus CMP completion), so it was reverted too. Net result: no signature popup. The user can call `vim.lsp.buf.signature_help()` manually if they want one; auto-trigger is left disabled. Revisit when a less intrusive engine (CMP docs panel only, or virtual-text hint inside CMP item) is wanted.

Treesitter-context (`nvim-treesitter/nvim-treesitter-context`) crashed with `attempt to call method 'range' (a nil value)` from `get_parent_langtrees â†’ parse()` â€” `multiwindow = true` makes it try to render context for every visible window, including LSP hover floats whose parser tree isn't fully populated. Fixed defensively in `lua/plugins/treesitter-context/init.lua`: `multiwindow = false` removes the multi-window scan path, and `on_attach = function(buf) return vim.bo[buf].buftype == "" end` skips non-file buffers as a second line of defense. Drop the `on_attach` if `multiwindow=true` becomes stable upstream. Plan and verification checklist live at `docs/lsp-float-ux-plan.md`.
